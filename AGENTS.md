# AGENTS.md

Single source of agent instructions for this repository. `CLAUDE.md` and `.github/copilot-instructions.md` are symlinks to this file. See [ARCHITECTURE.md](ARCHITECTURE.md) for system design and [docs/agent/STEPS.md](docs/agent/STEPS.md) for the full step reference.

## Project Overview

Reusable Playwright + Gherkin E2E testing boilerplate with ~150 ready-made step definitions, MCP Test Engine for AI-driven browser sessions, and optional Syngrisi visual regression.

## Tech Stack

- Node 24 (`.nvmrc`), TypeScript 5.9 (strict), yarn 1.x
- Playwright 1.57 + playwright-bdd 8.4 (Gherkin → generated specs)
- Syngrisi 3.5 (visual regression, requires local MongoDB)
- MCP SDK 1.21 (AI agent browser sessions)
- envalid (config), winston (logging), zod (validation)

## Common Rules

- Answer in the language you are asked in.
- Always write comments and debug messages in your code in English.

## NEVER

- NEVER edit files in `.features-gen/` — they are generated by `bddgen`.
- NEVER use `npm` — this project uses `yarn` only.
- NEVER commit `.env` or secrets; use `.env.example` for new variables.
- NEVER hardcode site URLs in features — use `<baseUrl>` (resolved from `E2E_BASE_URL`).
- NEVER modify Syngrisi baselines or `.snapshots-images/` manually — accept baselines via the Syngrisi UI.

## ALWAYS

- ALWAYS run `yarn bddgen` after changing `features/` or `steps/` before running tests or test-engine sessions.
- ALWAYS run `yarn type-check` before finishing a task.
- ALWAYS search [docs/agent/STEPS.md](docs/agent/STEPS.md) for an existing step before writing a new one; add new steps only to `steps/domain/`.
- ALWAYS set `SYSTEM_THREAD` env var for manual test-engine CLI flows.

## ASK FIRST

- Before deleting or renaming any step in `steps/common/` (steps are the public API of this boilerplate).
- Before changing the test-engine CLI command surface (`start`, `step`, `batch`, `status`, `shutdown`).
- Before editing `support/mcp/server.ts` or `support/mcp/bridge.ts`.

## Design & Coding Principles

- Keep it simple. No overengineering, but preserve required functionality.
- Solve problems the simplest way possible.
- Reuse existing entities. Don't create new ones without necessity.
- Trust frameworks. Don't add unnecessary wrapper layers.
- No abstraction layers without clear need.

## Project Structure

```
.
├── config.ts                # Environment configuration (envalid)
├── playwright.config.ts     # Playwright BDD configuration
├── steps/
│   ├── common/              # Universal step definitions (do not break their texts)
│   ├── domain/              # App-specific step definitions (add new steps here)
│   └── helpers/             # Locator builders, assertions, template engine
├── features/                # Gherkin feature files
├── scripts/                 # Maintenance scripts (steps doc generator)
├── support/
│   ├── fixtures/            # Playwright fixtures (app-server, test-data, test-engine, syngrisi)
│   ├── params.ts            # BDD parameter types (role, ordinal, condition)
│   ├── mcp/                 # MCP server/bridge/test-engine (has own AGENTS.md)
│   ├── demo/                # Demo utilities (highlight, speech, progress)
│   ├── lib/                 # Logger
│   └── utils/               # waitFor, hasTag, sleep
└── docs/agent/              # Agent documentation guides
```

## Key Abstractions

- **Fixtures merge**: `support/fixtures/index.ts` merges all fixtures via `mergeTests` and exports `Given/When/Then` — import them from `@fixtures` in step files.
- **BDD params**: `{role}`, `{ordinal}`, `{condition}` custom parameter types live in `support/params.ts`.
- **Template engine**: `test-data` fixture resolves `<baseUrl>` and stored values inside step arguments.
- **MCP code uses relative imports** (no tsconfig aliases) — the bridge runs without tsconfig path resolution.

## Path Aliases (tsconfig.json)

- `@config` → `config.ts`
- `@fixtures` → `support/fixtures/index.ts`
- `@fixtures/*` → `support/fixtures/*`
- `@helpers/*` → `steps/helpers/*`
- `@utils/*` → `support/utils/*`
- `@lib/*` → `support/lib/*`
- `@params` → `support/params.ts`
- `@logger` → `support/mcp/utils/logger.ts`

## Commands

```bash
yarn test              # Run all E2E tests
yarn test:smoke        # Run @smoke tagged tests
yarn test:headed       # Run tests in headed mode
yarn test:mcp          # Start MCP server for AI agents
yarn test:mcp:headed   # Start MCP server in headed mode
yarn test:examples     # Run @example tests
yarn test:visual       # Run stable Syngrisi visual examples
yarn test:visual:failing # Run intentionally failing Syngrisi example
yarn sy                # Start Syngrisi visual testing server
yarn steps:docs        # Regenerate docs/agent/STEPS.md from step definitions
yarn lint              # Biome lint
yarn format            # Biome format (write)
yarn verify            # Biome lint + format check (named 'verify' to avoid yarn's built-in 'check')
yarn test:unit         # Pure-logic unit tests (helpers), no browser
yarn type-check        # TypeScript type checking
yarn kill              # Kill all test processes
```

## Visual Testing (Syngrisi)

Visual regression testing via `@syngrisi/playwright-sdk`.

```bash
yarn test:visual
```

Requires MongoDB on `127.0.0.1:27017` (see README "Prerequisites"). Syngrisi autostarts from the fixture when it is not reachable. Default UI is `http://localhost:5566/`; logs are written to `logs/syngrisi/syngrisi.log`.
Config in `.env`: `SYNGRISI_BASE_URL`, `SYNGRISI_DB_URI`, `SYNGRISI_PROJECT`, `SYNGRISI_BRANCH`.
Set `DISABLE_VISUAL_CHECKS=true` to skip visual checks globally.
Tag tests with `@no-visual` to skip per-test.

## Browser Skill (test-engine CLI)

Interactive browser automation via `/browser` skill or direct CLI:

```bash
export SYSTEM_THREAD=manual-browser
npx tsx support/mcp/test-engine-cli.ts start my-session --headed
npx tsx support/mcp/test-engine-cli.ts step "I open site \"<baseUrl>\""
npx tsx support/mcp/test-engine-cli.ts step "I analyze current page"
npx tsx support/mcp/test-engine-cli.ts shutdown
```

Rules:
- Always set `SYSTEM_THREAD` for manual flows
- Run `yarn bddgen` before scenario work
- Use `step` for single actions, `batch` for 2+ sequential steps
- Use `status --json` to inspect session health

## Definition of Done (DoD) Checklist

Before marking a task as complete, conduct a final self-review using the following checklist:

1. **Test Coverage & Execution:** Have all necessary tests for the new functionality been written and successfully executed?
2. **Documentation:** Is the documentation updated to reflect the new changes?
3. **Code Review:** Have you reviewed all modified files for syntax errors, leftover debug code, and coding standards?
4. **Smoke Testing:** Have you run a smoke test to ensure the main functionality is working and no regressions were introduced?
